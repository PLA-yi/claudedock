# 51-02 QUAL-02 — `verifyLeakBlocked` 多目标参数化

## Goal

把 `internal/network/verify.go::verifyLeakBlocked` 从「单 target `1.1.1.1:80`」
改造成「多 IP × 多端口矩阵 + 任一连通 = fail」。默认值与 Phase 46
`DefaultDenyMatrix` 4 target 对齐：

```
1.1.1.1:80
8.8.8.8:443
9.9.9.9:443
169.254.169.254:80
```

接口契约保护：旧签名 `verifyLeakBlocked(ctx, prefix, *VerifyResult)` 保留；
内部委托给 `verifyLeakBlockedMulti(ctx, prefix, targets, *VerifyResult)`，默认
传入包级常量 `defaultLeakTargets`。

## Files

- `internal/network/verify.go`（改）—— 新增 `defaultLeakTargets` 常量切片 +
  `verifyLeakBlockedMulti` 内部实现；旧 `verifyLeakBlocked` 包装它。
- `internal/network/verify_test.go`（改）—— 新增 4-pass / 任一连通 fail / 全 fail
  的纯函数单测。

## Steps

1. 在 `verify.go` 新增类型 `type leakTarget struct{ Host string; Port int }` 与
   常量切片 `defaultLeakTargets = []leakTarget{ {Host:"1.1.1.1",Port:80},
   {Host:"8.8.8.8",Port:443}, {Host:"9.9.9.9",Port:443},
   {Host:"169.254.169.254",Port:80} }`。
2. 新增 `verifyLeakBlockedMulti(ctx, prefix, targets, *VerifyResult)`：
   - 并发对每个 target 跑 `timeout 3 bash -c 'echo >/dev/tcp/HOST/PORT'`；
   - 任一 exit 0（连通） → `LeakBlocked=false`，把第一个连通的 target 写入
     `LeakTarget`；
   - 全部非 0 → `LeakBlocked=true`，`LeakTarget` 写入「所有 target 都 blocked」
     占位字符串 `"all_blocked"`（或保持空 + 锁端口语义）。
3. 重写旧 `verifyLeakBlocked` 为：先 `result.LeakTarget = "1.1.1.1:80"` 保留对
   `TestVerifyNetworkIntegrity_LeakTargetSet` 的兼容；然后调用 `Multi` 函数。
   关键：在 `LeakBlocked=true` 路径 `LeakTarget` 保持原 `"1.1.1.1:80"` 默认值
   （兼容既有断言 `result.LeakTarget != ""`）；在 fail 路径覆盖为真实泄漏 target。
4. 单测：
   - `TestVerifyLeakBlockedMulti_AllBlocked`：4 个 target 全 err → `LeakBlocked=
     true`。
   - `TestVerifyLeakBlockedMulti_OneLeaked`：第 2 个 target 返回 nil → `LeakBlocked=
     false`，`LeakTarget="8.8.8.8:443"`。
   - `TestVerifyLeakBlockedMulti_AllLeaked`：4 个 target 全 ok → `LeakBlocked=
     false`，`LeakTarget` 取第一个（确定性）。
   - `TestDefaultLeakTargets_LockedContract`：长度 = 4，依次断言 4 行 host/port。

## Acceptance

- `go build ./...` + `GOOS=linux go build ./...` PASS。
- `go test ./internal/network/... -count=1` 全绿（含新增 4 个单测）。
- 既有 `TestVerifyNetworkIntegrity_LeakTargetSet` 不需要修改即过（默认
  `LeakTarget="1.1.1.1:80"`）。

## Risk & Mitigation

| Risk | Mitigation |
|------|-----------|
| 并发 4 个 nsenter 同时跑可能在 fake nsenterRunner 单测下产生 race 输出。 | fake nsenterRunner 仅返回固定值（不持有 state）；并发对 fake 安全。 |
| `LeakTarget` 字段语义微变（多 target 模式下到底写哪一个）。 | 保留默认值 `"1.1.1.1:80"`；fail 路径覆盖为「第一个连通 target」；既有 e2e 用例都只断言 `LeakBlocked` bool，对 `LeakTarget` 字符串无 strict 匹配。 |
| 169.254.169.254 IMDS target 与 Phase 49 LEAK-05 重叠。 | 这是预期：51-05 落地新 nft `linklocal-drop` 后，IMDS target 应被 nft 显式 drop，与 verify.go 多 target 检测同向。 |
| 影响 51-01？ | 不动 verifyEgressIP；无交叉。 |
| 影响 51-09？ | 不影响 admin_bindings；无交叉。 |
| 影响 e2e 用例？ | `VerifyResult` 字段语义不变；行为契约维持。 |
