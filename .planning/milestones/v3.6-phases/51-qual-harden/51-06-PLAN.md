# 51-06 QUAL-06 — worker `--cap-drop NET_RAW` + 删 SYS_ADMIN

## Goal

闭 Phase 49 GAP-1：worker 容器目前启动参数（`internal/runtime/tasks/worker.go::
buildCreateArgs`）显式 `--cap-add NET_ADMIN` + `--cap-add SYS_ADMIN`，且未显式
`--cap-drop NET_RAW`（docker 默认含 NET_RAW）。Phase 49 LEAK-06 / LEAK-08 用例
据此期望 `/proc/1/status` 的 `CapEff` / `CapBnd` 不含 NET_RAW / NET_ADMIN /
SYS_ADMIN。

**改造方向**（按 CONTEXT §Area 4「折中」决策）：
1. 删 `--cap-add SYS_ADMIN`（worker 业务侧不需要）。
2. 显式追加 `--cap-drop NET_RAW`（docker 默认带；显式 drop 才能去掉）。
3. **NET_ADMIN**：grep 源码确认 host-agent / sing-box 启动路径是否真的依赖
   namespace 内 `CAP_NET_ADMIN`。
   - 若依赖（如 `netlink.LinkSetUp` / `nftables.Flush` 在 worker netns 内执行）：
     **保留 `--cap-add NET_ADMIN`**，在 SUMMARY 明确说明 fixture
     `proc_status_clean.txt` 期望需要相应放宽（NET_ADMIN 保留时 LEAK-08 期望需
     重新评估，但 e2e fixture 不在本 plan 范围内）。
   - 若不依赖：一并删除 `--cap-add NET_ADMIN`。
4. fixture / 用例不动（CONTEXT 锁定）。

## Files

- `internal/runtime/tasks/worker.go`（改）—— 行 217-218 附近修改 `args` 切片。
- `internal/runtime/tasks/worker_test.go`（若存在；新增或扩展）—— 断言
  `buildCreateArgs` 输出不含 `SYS_ADMIN` + 含 `--cap-drop NET_RAW`。

## Steps

1. **决策点：NET_ADMIN 留还是删**。grep `internal/network/`：
   - `nftables.Flush` 在 worker netns 执行：调用方在 host netns 里通过
     `nftables.WithNetNSFd(int(containerNS))` 进入 worker netns 操作。这个 netlink
     操作需要 host netns 中持有 CAP_NET_ADMIN（host-agent 进程），与 worker
     容器是否带 CAP_NET_ADMIN 无关。
   - `netlink.LinkAdd` / `LinkSetUp` 在 `InjectManagementVeth` 中通过 `runtime.
     LockOSThread + netns.Set(containerNS)` 切到 worker netns 操作 —— 同样靠
     host-agent 进程 cap，不靠 worker 容器内 cap。
   - sing-box 在 worker netns 内运行（参考 singbox_provider_linux.go），sing-box
     `tun` 设备创建需要 `CAP_NET_ADMIN`。
   - **结论**：sing-box `tun` 创建路径需要 worker 容器 netns 内有 NET_ADMIN
     （sing-box 进程通过 `nsenter -n` 进入 netns 执行）。**保留
     `--cap-add NET_ADMIN`**。
2. **修改 `buildCreateArgs`**：
   - 删除 `"--cap-add", "SYS_ADMIN"`。
   - 在 `--cap-add NET_ADMIN` 之后追加 `"--cap-drop", "NET_RAW"`。
   - 保留 `--cap-add NET_ADMIN`，并在 worker.go 顶部加 doc comment 说明
     「NET_ADMIN 因 sing-box 在 worker netns 内创建 tun 设备而必须保留；
     SYS_ADMIN 已删除（业务不依赖）；NET_RAW 显式 drop（去掉 docker 默认
     CAP_NET_RAW，闭 Phase 49 LEAK-06 SOCK_RAW 攻击面）」。
3. **新增单测**：在 `internal/runtime/tasks/worker_test.go`（若文件不存在则
   新建）添加 `TestBuildCreateArgs_CapabilitiesLocked`：
   - 构造一个最小合法 `agentapi.HostActionRequest`（含 EntryPassword）；
   - 调 `(*Worker).buildCreateArgs(...)`；
   - 断言返回 args 切片中：
     - 含 `--cap-add NET_ADMIN`
     - 不含 `--cap-add SYS_ADMIN`（locked）
     - 含 `--cap-drop NET_RAW`（locked）

## Risk & Mitigation

| Risk | Mitigation |
|------|-----------|
| 删 SYS_ADMIN 后某个 worker 业务路径（如 fuse mount）受影响。 | 已保留 `--device /dev/fuse + --security-opt apparmor=unconfined`，fuse 实际不需要 SYS_ADMIN（容器内 fusermount setuid root 即可）；如 CI 出现 fuse 故障，回滚此条改动。 |
| NET_ADMIN 保留与 Phase 49 LEAK-08 fixture（`proc_status_clean.txt` 期望不含 NET_ADMIN）矛盾。 | CONTEXT §Area 4 明确允许折中保留 NET_ADMIN；SUMMARY 中记录该决策；fixture 修订/调整属后续 Phase 49 二次工作（不阻塞本 phase）。**已在 SUMMARY 明确披露。** |
| docker `--cap-drop NET_RAW` 与 `--cap-add NET_ADMIN` 同时使用是否生效？ | docker 支持同时 add / drop；NET_RAW drop 后 SOCK_RAW 拒绝，NET_ADMIN 仍可用 —— 符合预期。 |
| 影响 51-01..05？ | 仅改 worker.go；不交叉。 |
| 影响 51-09？ | 不交叉。 |
| 影响 e2e 用例？ | Phase 49 fixture 锁定不动；本 plan 不修改 e2e；Linux runner 实际跑通需 fixture 后续修订。 |
