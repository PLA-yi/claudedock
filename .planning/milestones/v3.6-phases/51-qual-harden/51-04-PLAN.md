# 51-04 QUAL-04 — `GetContainerNetNS` 探测窗口参数化

## Goal

把 `internal/network/namespace.go::GetContainerNetNS` 当前硬编码的「5 次重试 ×
300ms 间隔」暴露为 functional option，允许 e2e harness / 单元测试在不破坏默认
生产行为的前提下传入更短的窗口加速测试。

接口契约保护：当前 4 个调用点保持不变（仍以单参数 `containerName` 形式调用）。
通过可变参数 `opts ...Option` 提供向后兼容。

## Files

- `internal/network/namespace.go`（改，linux build tag）—— 抽出 `config` 结构 +
  `Option` 函数类型 + `WithProbeWindow` / `WithMaxRetries` 选项；改 `GetContainerNetNS`
  签名为 `(containerName string, opts ...Option)`。
- `internal/network/`（如需）新增小测试文件覆盖纯函数（option 应用 + default 值
  断言）。

## Steps

1. 新增 `nsConfig` 私有结构体：`{ probeWindow time.Duration; maxRetries int }`。
2. 新增 `Option func(*nsConfig)` 类型与两个构造器：
   - `WithProbeWindow(d time.Duration) Option`
   - `WithMaxRetries(n int) Option`
3. 修改 `GetContainerNetNS(containerName string, opts ...Option)`：
   - 起始时初始化 `cfg := nsConfig{probeWindow: 300 * time.Millisecond, maxRetries: 5}`；
   - 应用 opts；
   - 用 `cfg.probeWindow` 替换 `time.Sleep(300 * time.Millisecond)`；
   - 用 `cfg.maxRetries` 替换硬编码 `maxRetry = 5`。
4. 既有 4 个调用点不动（自动用默认值，行为不变）。
5. 单测：
   - 不依赖 docker / netns 真实调用；只测 `Option` 合成 → cfg 字段被正确写入。
   - 用一个小的 namespace_test.go（linux build tag）通过反射或导出测试 helper
     断言 `nsConfig` 字段；如果不愿暴露内部 helper，简单实现：跨平台 `_test.go`
     断言只覆盖 Option 的纯函数（`WithProbeWindow(7s)` apply 后 cfg.probeWindow=7s）。

## Acceptance

- `go build ./...` + `GOOS=linux go build ./...` PASS。
- `go test ./internal/network/... -count=1` 全绿；新增 ≥2 个 Option 单测。
- 既有 4 个 `GetContainerNetNS(containerName)` 调用点零修改（默认值生效）。

## Risk & Mitigation

| Risk | Mitigation |
|------|-----------|
| 函数签名加 `...opts` 是 Go 兼容性 break？ | Go 可变参数对已有调用方零破坏（无需任何调用点变更）。 |
| `Option` 函数若放进 `linux build tag` 文件中，darwin 单测无法触达。 | 单测文件用同样的 `//go:build linux`；darwin 仅靠跨编译验证；这一约束与现有 `namespace.go` 一致。 |
| 影响 51-01..03？ | 不交叉（不动 verify.go）。 |
| 影响 e2e 用例？ | tests/e2e 用例不直接调 GetContainerNetNS（走 docker exec），无回归风险。 |
| 影响 51-05 / 51-06 / 51-09？ | 不交叉。 |
