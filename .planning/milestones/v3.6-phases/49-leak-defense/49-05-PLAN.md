# Phase 49 Plan 05: LEAK-05 IMDS 阻断

## Goal

证实 worker 容器内访问云厂商 IMDS 端点必须失败：

- `curl --max-time 3 http://169.254.169.254/latest/meta-data/`（AWS / GCP / Azure 共享 169.254.169.254 入口）必须失败。
- `curl --max-time 3 http://169.254.170.2/v2/credentials/x`（ECS task IAM 角色）必须失败。
- nft `link-local drop` 规则（如存在）的 counter `>0`（命中标志）。
- host eth0 IPv4 抓包 BPF：`dst net 169.254.0.0/16 and src host <workerIP>` 必须 0 包。

`TestLeak_05_IMDS_BlockedByHostEth0`。

  - `*GoldenPath.CurlIMDS(ctx, url string) (*LeakProbeResult, error)`。

## Steps

1. **`*GoldenPath.CurlIMDS`**：
   - `docker exec <worker> curl -sS -o /dev/null -w '%{http_code}' --max-time 3 <url>`。
   - 解析：
     - exit 0 + http_code == "200"（含 IMDS 数据） → `Blocked=false`（实锤泄漏）。
     - exit 0 + http_code 4xx/5xx → `Blocked=true, Reason="imds_http_error"`（被代理/网关返回错而非真实 IMDS）。
     - exit 7 / 28 / stderr 含 `Connection refused` / `Connection timed out` → `Blocked=true`。

2. **主用例 `TestLeak_05_IMDS_BlockedByHostEth0`**：
   - 起 host eth0 tcpdump goroutine：BPF `dst net 169.254.0.0/16 and src host <workerIP>`，count=5，window=5s。
   - 串行 `g.CurlIMDS(ctx, "http://169.254.169.254/latest/meta-data/")` 与 `g.CurlIMDS(ctx, "http://169.254.170.2/v2/credentials/x")`，两次都必须 Blocked。
   - 等 tcpdump → packets。
   - 裁决：`tdRes.packets > 0 || any !Blocked` → t.Fatalf。
   - 复用 Plan 07 的 `ListNftRulesOnHost` + `ParseNftCounters` 选做 counter 命中校验（counter > 0 强证 nft 规则真接管；为 0 不一定 fail，因为可能没显式 link-local 规则但 default-deny 兜住，oracle 还是看抓包）。

3. **darwin 兜底**：build tag。

## Risk & Mitigation

| 风险 | 缓解 |
|------|------|
| 有些云 runner（GCP）真的在 169.254.169.254 上响应 200，导致 tcpdump 不在 *worker* eth0 上抓到 ——但本测在 docker 默认 bridge 上，worker 走 sb-tun0；只要 host eth0 不出现 worker→169.254 的包就符合 | tcpdump BPF `src host <workerIP>` 已限定包源；GCP host 自身的 169.254 流量不会命中。 |
| nft 当前是否有 *专门* `169.254.0.0/16 drop` 规则未确证（grep 未发现） | counter 命中是 nice-to-have；用例不强要求 counter>0，强要求只有 host eth0 0 包。**LEAK-07 单独验证 nft 规则形态**。 |
| worker 镜像无 curl | grep 已确认 managed-user 镜像装 curl。 |
