# 51-05 QUAL-05 — nftables 全规则加 counter + 169.254/16 显式 drop

## Goal

两件事一起做：

1. **全规则加 counter 表达式**：把 `internal/network/worker_firewall_linux.go`
   与 `internal/network/firewall_helpers.go` 中所有 `nft AddRule` 调用插入
   `&expr.Counter{}`（不破坏行为，仅观测）。Phase 49 引入的 `ParseNftRules` /
   `ParseNftCounters` 单测据此能在 `nft list ruleset` 输出中读出每条规则的命中
   数。
2. **新增 IPv4 `169.254.0.0/16` 显式 drop counter 规则**（闭 Phase 49 GAP-2）：
   在 worker netns nft `output` 链白名单规则**之前**插入显式
   `ip daddr 169.254.0.0/16 counter drop comment "linklocal-drop"`，让
   LEAK-07 用例从 fail → pass。

## Files

- `internal/network/firewall_helpers.go`（改，linux）—— 新增
  `addIPDaddrDropRule(conn, table, chain, cidr string)` helper + `buildIPDaddrDropExprs`
  纯函数；既有 `buildOif*` / `addOif*` / `addIif*` 系列规则统一插入 `&expr.Counter{}`。
- `internal/network/worker_firewall_linux.go`（改，linux）—— 在
  `applyWorkerIPv4Rules` 的 OUTPUT 链构造中，**先**追加
  `addIPDaddrDropRule(conn, table, outputChain, "169.254.0.0/16")`，**再**放
  whitelist accept / DNS accept / sing-box bypass 等。
- `internal/network/firewall_helpers_test.go`（改）—— 新增 `buildIPDaddrDropExprs`
  纯函数单测 + counter 表达式存在性断言。

## Steps

1. **新增 `buildIPDaddrDropExprs`**：
   ```
   meta nfproto ipv4
   ip daddr <cidr>
   counter
   drop
   ```
   配合 `expr.Bitwise` / payload 实现 `/16` mask 匹配，与 `addOifDstIPAcceptRule`
   的 `/32` 匹配区分（后者直接比 4 字节）；`/16` 仅匹配高 16 位。
2. **新增 `addIPDaddrDropRule`** wrapper：调 `conn.AddRule` 写入；comment 字段
   用 nft userdata（参考 `github.com/google/nftables` 文档 `Rule.UserData`）写入
   `"linklocal-drop"`。
3. **批量插入 counter**：扫描 `firewall_helpers.go` 中所有 `Exprs: []expr.Any{...}`
   列表，在每条 verdict 之前插入 `&expr.Counter{}`。重点函数：
   - `addOifAcceptRule` / `addIifAcceptRule`
   - `addOifCtEstablishedRule` / `addIifCtEstablishedRule`
   - `addIifTCPDportAcceptRule`
   - `buildOifSkuidIPPortAcceptExprs`
   - `buildOifNameAcceptExprs`
   - `buildOifNamedSetMatchAcceptExprs`
   - `buildOifUDPDportDropExprs`
   - `buildLogDropExprs` 已经有 counter，不动。
4. **插入 link-local drop**：
   在 `applyWorkerIPv4Rules` 函数体内，**OUTPUT 链 lo / ESTABLISHED accept 之后、
   gwIP / DNS / whitelist accept 之前**，调
   `addIPDaddrDropRule(conn, table, outputChain, "169.254.0.0/16")`。
   位置选择理由：lo / ESTABLISHED 允许放在最前以保持原始性能基线；link-local drop
   在所有 accept 规则之前，覆盖即便后续 whitelist 出现 169.254 entry 也兜底 drop。
5. **更新单测**：
   - `TestBuildIPDaddrDropExprs_LinkLocal`：断言 exprs 序列依次为
     `[Meta nfproto, Payload(daddr), Bitwise(/16), Cmp(=), Counter, Verdict(Drop)]`。
   - `TestBuildLogDropExprs_HasCounter` 已存在，保留。
   - 既有 helpers 单测如果断言 exprs 长度等具体数字，需同步调整长度（+1 for counter）。

## Risk & Mitigation

| Risk | Mitigation |
|------|-----------|
| 大量 helper 加 counter 改动可能触发既有单测断言（exprs 长度 / 索引）。 | 单测文件全量过一遍；新增 counter 永远是 `verdict` 之前的「最后一个 non-verdict」expr，索引向前 push 1；调整对应断言。 |
| `/16` mask 表达式比 `/32` 复杂（需 Bitwise）。 | 参考 `github.com/google/nftables` 文档 + repo 内现成 mask 范式；用 `expr.Bitwise{Len:4, Mask:[0xff,0xff,0x00,0x00], Xor:[0,0,0,0]}`；先与 `0xffff0000` 做 AND，再与 `169.254.0.0` 比相等。 |
| Linux runner LEAK-07 是否真的 → PASS 取决于 fixture `nft_ruleset_with_link_local_drop.txt` 与实际 nft 输出格式是否一致。 | fixture 是「修复后预期」，本 plan 落地后实际 `nft list ruleset` 输出会包含 `ip daddr 169.254.0.0/16 counter drop comment "linklocal-drop"` 行，与 fixture 行式 grep 一致。 |
| Counter 加多了影响性能？ | nftables counter 是内核 inline counter，每包 ~2 cycles 开销，可忽略。 |
| 影响 51-01..04？ | 不交叉（不动 verify.go / namespace.go / admin_bindings）。 |
| 影响 51-06 / 51-09？ | 不交叉。 |
| 影响 e2e 用例？ | Phase 49 LEAK-07 fixture 已经反映「修复后」预期，不动；LEAK-05 IMDS 在 51-05 后由「链末 sbfw-drop 兜住」→「显式 linklocal-drop 命中」，counter 命中数 ≥1，仍 PASS。 |
